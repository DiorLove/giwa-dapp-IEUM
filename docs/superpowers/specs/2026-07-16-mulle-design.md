# 물레 MULLE — 설계 문서

날짜: 2026-07-16
상태: 승인됨
목표: GIWA GASOK 빌더 프로그램 지원 (마감 2026-07-31) — 테스트넷에서 동작하는 MVP + 지원서 제출

## 1. 한 줄 정의

한국의 전통 금융 프리미티브 '계(契)'를 GIWA 체인 스마트 컨트랙트로 재구현한 온체인 계모임 dApp.
태그라인: "계주는 컨트랙트, 신뢰는 체인에. 돌려라, 목돈이 온다."

## 2. 배경 및 전략

### GASOK 프로그램 핵심
- 주최: Upbit × GIWA (두나무 이더리움 L2, OP Stack, 1초 블록타임)
- 혜택: 팀당 최대 $100K (데모데이 우승 $20K + KPI 달성 최대 $80K), 컨트랙트 감사, KBW 데모데이(10월), GIWA 월렛 인앱 탑재 기회
- 일정: 7/31 신청 마감 → 8/14 발표 → MVP/테스트넷(Phase 2) → 프로덕트화(Phase 3) → 성장(Phase 4)
- 심사 기준 6가지: 왜 GIWA인가 / 독창성 / 실행 가능성 / 시장성 / 팀 역량 / GIWA 월렛 탑재 가능성

### 지원 트랙
- 주력: GIWA-Native Ideas
- 복수 신청: Mass Adoption

### 왜 GIWA인가 (지원서 핵심 논리)
1. 계는 한국 금융문화 그 자체 — 한국 체인에서만 나올 수 있는 앱
2. 두나무 KRW 스테이블코인 로드맵의 킬러 유스케이스 (스테이블 출시 순간 실생활 저축 앱)
3. UP.ID 기반 온체인 신용이력 (계 완주 기록 = 신용 자산)
4. GIWA 월렛 인앱 "저축 탭" 최적 후보
5. 1초 블록타임 = 납입/지급 즉시 확정 UX

### 해결하는 문제
계의 3대 신뢰 문제를 코드로 제거:
- 계주 먹튀 (컨트랙트가 계주 대체)
- 순번 조작 (온체인 제비뽑기 또는 전원 서명 동의)
- 장부 분쟁 (모든 납입/지급이 온체인 기록)

핵심 난제인 "먼저 탄 사람의 납입 중단"은 보증금 슬래싱으로 해결 (하이브리드 2티어의 1단계).

## 3. 팀 상황 (설계 제약)

- 1인 빌더, 일반 개발 경험 있음, 온체인 개발 첫 경험 (AI 도구 활용)
- 따라서: 컨트랙트는 검증된 단순 패턴만, 백엔드 없는 순수 dApp, 범위 최소화

## 4. 제품 구조 (하이브리드 2티어 + 확장 로드맵)

- **티어 1 친목계 (MVP)**: 지인 초대 기반, 보증금 옵션, 완주 기록 축적
- **티어 2 오픈계 (Phase 3 로드맵)**: 비지인 매칭, 필수 보증금 + 순번 경매(낙찰계 = 온체인 금리 시장)
- **확장 (Phase 4 로드맵)**: "모임 금고" 프리미티브로 일반화 → 모임 회비, 팬덤 서포트 모금, 경조사비

## 5. MVP 범위 (7/31까지)

### 포함
1. **계 개설**: 인원(3~12명), 회당 납입액(mKRW), 납입 주기, 순번 방식 선택, 보증금 옵션(0~2회분)
2. **순번 방식 (듀얼 모드)**:
   - 🎲 온체인 제비뽑기: 계 시작 시 blockhash 기반 랜덤으로 순번 공개 추첨
   - 📝 계주 지정: 개설자가 순번 제안 → 전원 지갑 서명 동의 시 시작
3. **초대 링크 참여**: 링크 → 지갑 연결 → 보증금 예치(옵션) → 참여. 카톡 공유 최적화
4. **회차 진행**: 납입 기간 내 mKRW 납입 → 마감 시 해당 순번자에게 지급(pull 방식 claim). 미납 시 보증금 자동 차감
5. **완주/파탄 처리**: 완주 시 보증금 반환 + 온체인 완주 기록. 파탄 시 비례 정산 환불
6. **mKRW 파우셋**: 데모용 모의 원화 무료 발급

### 제외 (로드맵으로 명시)
낙찰계(경매), 오픈계 매칭, UP.ID 연동, 평판 시스템, 푸시 알림, 실물 KRW 스테이블 연동

## 6. 아키텍처

```
[모바일 웹 (Next.js + wagmi/viem)]
        │ 지갑 연결 (MetaMask / WalletConnect)
        ▼
[GIWA 테스트넷 (Sepolia)]
 ├─ MockKRW.sol      — ERC-20 mKRW + 파우셋 (₩ 단위 데모)
 ├─ MulleFactory.sol — 계 생성, 계 목록 조회
 └─ Mulle.sol        — 계 1개 = 컨트랙트 인스턴스 1개
```

### 컨트랙트 설계
- Mulle 상태머신: `모집중 → 진행중 → 완주 | 파탄`
- 회차 마감: 누구나 호출 가능한 `settle()` (호출자 소액 인센티브) — 서버리스 자동화
- 지급: pull 방식 — 수령액을 컨트랙트에 적립하고 본인이 `claim()`
- 랜덤: MVP는 blockhash 기반 (친목계 소액에 충분). 오픈계 단계에서 commit-reveal 업그레이드 명시
- 개발 스택: Foundry (테스트 포함), OpenZeppelin ERC-20

### 프론트
- Next.js 모바일 퍼스트 웹, wagmi/viem
- 핵심 3화면: 계 개설 / 초대·참여 / 계 현황(물레 회전 UI — 순번 시각화)
- 백엔드 없음: 컨트랙트 이벤트를 클라이언트에서 직접 조회

## 7. 엣지 케이스

| 상황 | 처리 |
|---|---|
| 미납 | 보증금에서 자동 차감. 보증금으로 미납액을 메꿀 수 없으면 자동 파탄 처리 (투표 메커니즘은 오픈계 단계 로드맵) |
| 모집 미달 | 모집 마감일까지 정원 미달 시 전액 환불 후 종료 |
| 파탄 | 컨트랙트 잔액 + 몰수 보증금을 **미수령 멤버**에게 각자 총 납입액 비례로 분배. 기수령자는 남은 보증금만 반환 |
| 수령자 지갑 문제 | pull 방식 claim으로 회피 |
| 순번 지정 동의 거부 | 전원 서명 미완 시 시작 불가, 개설자가 순번 수정 재제안 가능 |

## 8. 16일 실행 계획 (7/15 → 7/31)

- D1–5: 컨트랙트 개발 + Foundry 테스트 → GIWA 테스트넷 배포
- D6–11: 프론트 3화면 + 물레 회전 UI
- D12–13: 지인 2~3명과 테스트넷에서 계 1바퀴 완주 → 트랜잭션 기록 확보 (지원서 증거)
- D14–16: 지원서 작성 + 90초 데모 영상 + 제출 (버퍼 포함)

## 9. 지원서 승부 포인트

1. "이미 돌아갑니다" — 테스트넷 컨트랙트 주소·실사용 트랜잭션 링크 첨부
2. 성장 서사 — 친목계 → 낙찰계(DeFi 금리 시장) → 모임금고(회비·팬덤·경조사비)
3. KPI 구조 — 매달 반복 납입 = TVL·거래량·리텐션이 제품 구조에서 자동 발생
4. 월렛 탑재 비전 — "GIWA 월렛 안의 저축 탭"
5. 네이밍 — 물레(MULLE): 기와·보자기·도장과 같은 전통 사물 네임 패밀리 합류, 회전(Rotating Savings) 메커니즘 형상화
