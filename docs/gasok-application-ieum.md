## [4/12] 지원 동기 (500자 이내)

**제출 답변 (약 480자):**

> 한국인의 목돈은 늘 '사람의 손'을 거칠 때 사고가 납니다. 전세금은 집주인을 거치며 전세사기가 되고, 이사 잔금은 날짜가 하루만 어긋나도 거래가 깨지며, 곗돈은 계주 먹튀로 사라집니다. 저는 이 세 가지가 모두 하나의 문제 — 목돈이 중개자의 손에 머무는 시간 — 라고 봤고, 그것이 정확히 스마트 컨트랙트가 풀도록 설계된 문제라 생각했습니다. 이음은 신규 세입자의 전세금을 온체인에 락하고, 정산일에 단 하나의 트랜잭션으로 기존 세입자 보증금 반환과 집주인 잔금 지급을 동시에 실행합니다. 돈이 사람 손에 머무는 순간이 0이 됩니다. 이사 날짜의 며칠 공백은 담보가 눈에 보이는 브리지 풀이 잇고, 같은 에스크로 프리미티브로 계주 없는 계모임까지 확장됩니다. 이 제품은 GIWA에서만 완성됩니다. 두나무의 KRW 스테이블코인이 나오는 순간 이음은 즉시 실생활 전세·저축 앱이 되고, UP.ID는 거래 당사자 검증과 온체인 신용이력을 만들며, GIWA 월렛에 가장 자연스럽게 들어갈 '내 목돈을 지키는 탭'이기 때문입니다. 기와·보자기·도장처럼 '이음'이라는 이름도 GIWA의 전통 사물 세계관을 따랐습니다.

## [10/12] 기술 문서 / 원페이저 링크

**원페이저에 넣을 내용:**

> ## 이음 IEUM — 기술 원페이저
> **아키텍처**: 백엔드 없는 순수 dApp. Next.js(wagmi/viem) → GIWA Sepolia 직접 통신. 문서 원본은 온체인에 올리지 않고 keccak256 해시만 앵커(위·변조 증명), 원본은 로컬 보관·당사자만 열람.
> **컨트랙트(6종)**: MockKRW(교체 가능한 모의 원화) / JeonseFactory·JeonseEscrow(전세 1건 = 인스턴스 1개) / BridgePool(ERC-4626形 유동성) / MulleFactory·Mulle(계 1개 = 인스턴스 1개)
> **전세 상태머신**: 자금 대기 → 락 완료 → 정산 완료 | 취소
> **핵심 메커니즘**:
> - 원자적 연쇄 정산: 정산일에 누구나 호출 가능한 settle() 한 번으로 보증금 반환 + 집주인 잔금이 같은 트랜잭션에서 동시 확정 (한쪽만 지급되는 상태가 원천 불가능)
> - 브리지 풀: 다음 세입자의 전세금이 이미 락된 것을 컨트랙트가 확인한 뒤에만 보증금을 선지급(수수료 0.5%), 정산 시 자동 상환 = 담보가 보이는 초단기 대출. 수수료는 예치자 수익.
> - 문서 앵커: 계약서 등 서류의 keccak256 해시만 온체인 기록(서류 하이패스)
> - 계모임: 온체인 제비뽑기(blockhash 시드 Fisher–Yates) 또는 계주 제안+전원 서명 / 미납은 보증금 슬래싱 / pull 방식 claim
> - 자동화: settle() 호출 보상(0.1%)로 서버리스 회차 마감
> **알려진 한계와 계획**: blockhash 랜덤은 소액 친목계용 → 오픈계에서 commit-reveal 업그레이드 / 실KRW 스테이블 연동 전 감사 필수 / 감사 전 메인넷 미배포 원칙
